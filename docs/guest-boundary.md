# Guest Boundary

The Guest is untrusted. It may be an AI agent, Codex, Claude Code, Cursor agent, LLM runtime, or AI container. The Host is trusted and owns secrets, policy, execution context, redaction, and audit logs.

## Guest May Read

The Guest may read `privenv.manifest.json` only.

The manifest is generated by the Host and contains safe capability metadata:

- capability IDs
- descriptions
- capability kind
- fixed command program and args when declared safe
- env variable names only
- `exposedToGuest: false`

The manifest must never contain raw env values, vault values, tokens, PII, SSH keys, OAuth tokens, or production-looking secrets.

## Guest May Send

The Guest may send `EffectRequest` JSON to the Host.

An `EffectRequest` selects a capability by `capabilityId`. It must not request raw secrets, arbitrary command strings, arbitrary env values, or unrestricted execution.

## Guest Must Never Read

The Guest must never read:

- `privenv.host.json`
- `.privenv/vault.json`
- `.privenv/audit.log.jsonl`
- `.env`
- `.env.*`
- Host-internal execution context objects

The Guest must never receive raw secret values.

## Forbidden Guest APIs

`privenv-host` must not provide these APIs to the Guest:

- `getSecret()`
- `getEnv()`
- `rawEnv()`
- raw secret-value APIs
- raw `.env` readers
- raw vault readers

## Host Owns

The Host owns:

- `privenv.host.json`
- `.privenv/vault.json`
- `.privenv/audit.log.jsonl`
- redaction policy and redaction execution
- capability resolution
- vault lookup
- Host-internal execution context
- protected run and explicit demo-run behavior

## Flow

```text
Guest
  -> EffectRequest
Host
  -> redaction
  -> audit
  -> redacted EffectResponse
Guest
```

The boundary is simple: Guests never read secrets directly. Guests request effects executed by the Host.
