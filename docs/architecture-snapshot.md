# Architecture Snapshot

This snapshot records the current architecture direction for `privenv-host` without inspecting sibling repositories.

## Current Repos

### privenv-host

`privenv-host` is the trusted Host-side component.

It currently owns:

- Host config: `privenv.host.json`
- local development vault: `.privenv/vault.json`
- audit logs: `.privenv/audit.log.jsonl`
- redaction
- static policy validation
- Host-internal execution context
- safe manifest generation
- `EffectRequest` handling
- redacted `EffectResponse` creation

The current runtime is simulate-only. Real command execution is not implemented.

### privenv-guest

Conceptually, `privenv-guest` is the untrusted Guest-side component.

By the local Host-side design contract, the Guest:

- reads the safe manifest generated by the Host
- lists capabilities from that manifest
- creates `EffectRequest` JSON
- does not read secrets or Host-owned files
- does not receive raw secret values
- has no transport implementation assumed by this Host snapshot yet

This document does not claim implementation details from `privenv-guest` beyond the explicit design contract documented locally in this repository.

## Current Protocol

`privenv-host` and `privenv-guest` must agree on:

- `EffectRequest`
- `EffectResponse`
- `RedactionSummary`
- safe manifest shape

For now, protocol documentation, local TypeScript types, and protocol fixtures are duplicated locally. A shared protocol package has not been created yet.

Until a shared package exists, changes to `docs/protocol.md` must be mirrored manually through an explicit copied spec, issue, PR text, release note, or other written coordination. A future candidate is `@privenv/protocol`.

Host work must not inspect `privenv-guest` to verify compatibility.

## Current Workflow

### Host-side workflow

```sh
privenv-host init
privenv-host validate
privenv-host doctor
privenv-host manifest > privenv.manifest.json
privenv-host run
privenv-host demo-run
```

`run` is protected normal mode and does not silently use fixture vault fallback. `demo-run` is explicit fixture/demo mode.

### Guest-side conceptual flow

```text
read privenv.manifest.json
list capabilities
create EffectRequest
send EffectRequest to Host by an external/stdin workflow later
receive redacted EffectResponse
```

The Host currently supports Phase 1 stdio JSON request/response at the CLI boundary, but this snapshot does not add Guest transport orchestration.

## Explicit Non-Goals

Current non-goals:

- no real command execution yet
- no `child_process` usage
- no external API gateway
- no Unix domain socket yet
- no passthrough mode in Host
- no automatic `.env` import
- no raw secret APIs such as `getSecret()`, `getEnv()`, or `rawEnv()`
- no production vault integration yet
- no browser/session/cookie capability handling in the current MVP

## Next Decision Points

Open decisions include:

- whether to create `@privenv/protocol`
- whether to implement stdio transport orchestration across Host and Guest
- whether to keep Host and Guest packages independent longer
- when and how to implement real execution, if ever
- how to model passthrough at an upper layer without adding passthrough mode to Host
- how to handle browser, session, or cookie capabilities in the future, outside the current MVP
